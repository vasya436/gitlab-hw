Опишите основные преимущества использования масштабирования методами:

1. Активный master‑сервер и пассивный репликационный slave‑сервер
Суть схемы: есть один активный сервер (master), который обрабатывает все операции записи и чтения. Пассивный сервер (slave) постоянно синхронизируется с master’ом, но не принимает запросы — он «на подхвате».

Основные преимущества:

Высокая доступность (High Availability, HA). Если master выходит из строя, система может автоматически или вручную переключиться на slave. Время простоя минимизируется — бизнес‑процессы быстро восстанавливаются.

Аварийное восстановление (Disaster Recovery). Пассивный slave служит актуальной резервной копией данных. В случае серьёзной аварии (сбой диска, ошибка администратора, атака) можно быстро развернуть новый master из реплики.

Снижение риска потери данных. Репликация гарантирует, что данные почти синхронно копируются на другой сервер. Даже если master повреждён, актуальная копия есть на slave.

Простота настройки и администрирования. Архитектура с одним пассивным slave проще в развёртывании и мониторинге, чем сложные кластеры. Меньше точек отказа и взаимосвязей для отслеживания.

Тестирование и откат. На пассивном slave можно тестировать обновления ПО, миграцию данных или новые конфигурации без влияния на работу основной системы. Если что‑то пошло не так, можно вернуться к старой версии.

2. Master‑сервер и несколько slave‑серверов
Суть схемы: один master отвечает за запись данных, а несколько slave‑серверов реплицируют их и принимают запросы на чтение.

Основные преимущества:

Масштабирование чтения (Read Scaling). Основная нагрузка от запросов на чтение (SELECT в SQL‑базах) распределяется между несколькими slave‑серверами. Это разгружает master и позволяет обрабатывать больше параллельных запросов.

Повышение производительности. Пользователи могут подключаться к ближайшему slave (географическое распределение), что снижает задержки (latency). Например, европейские клиенты читают с европейского slave, американские — с американского.

Балансировка нагрузки. Запросы на чтение можно распределять между slave’ами с помощью балансировщика нагрузки (например, HAProxy, pgpool‑II). Это предотвращает перегрузку одного сервера.

Отказоустойчивость. Если один из slave‑серверов выходит из строя, трафик перенаправляется на оставшиеся. Система продолжает работать без потери функциональности (при условии, что master жив).

Специализация серверов. Slave‑серверы можно настраивать под конкретные задачи:

один - для отчётности (тяжёлые аналитические запросы);

другой - для веб‑приложения (быстрые запросы);

третий - для резервного копирования (снимок данных без влияния на другие процессы).

Гибкость развёртывания. Slave‑серверы могут находиться в разных дата‑центрах или облаках, что повышает устойчивость к локальным сбоям и соответствует требованиям к хранению данных (data residency).

Нагрузочное тестирование.

Проверка отказоустойчивости.

Вертикальный шаринг (разбивка по таблицам):

Таблица пользователи - отдельный сервер (профиль, аутентификация, история заказов).

Таблица книги — отдельный сервер (каталог, описания, рейтинги).

Таблица магазины — отдельный сервер (локации, контакты, часы работы).
