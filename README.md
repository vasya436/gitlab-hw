# Домашнее задание к занятию  "Базы данных, их типы"


### Задание 1

Задание 1. СУБД

Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Приведите ответ в свободной форме.

Решение:

1.1. Бюджетирование и финансовая аналитика
Рекомендуемая СУБД: реляционная (PostgreSQL, Microsoft SQL Server, Oracle).

Обоснование:

Целостность данных. Реляционные СУБД поддерживают ACID‑транзакции (атомарность, согласованность, изоляция, долговечность), что критично для финансовых операций.

Чёткая структура. Табличная модель с жёсткой схемой идеально подходит для бюджетов, статей расходов, доходов и финансовых отчётов.

Сложные запросы. SQL позволяет строить сложные аналитические запросы, агрегации, оконные функции для прогнозирования.

Отчётность. Легко интегрируется с BI‑системами (Power BI, Tableau) для визуализации данных.

Аудит. Встроенные механизмы журналирования и контроля изменений.

1.2. Лендинги и CRM
Рекомендуемые СУБД:

Для лендингов: документо‑ориентированная (MongoDB, Couchbase).

Для CRM: документо‑ или реляционная (MongoDB или PostgreSQL).

Обоснование для лендингов:

Гибкость схемы. Лендинги могут собирать разные типы данных (формы обратной связи, заявки, подписки) — NoSQL позволяет хранить разнородные данные без жёсткой схемы.

Быстрая запись. Высокая производительность при большом потоке лидов.

Горизонтальное масштабирование. Легко масштабируется при росте трафика.

Обоснование для CRM:

Документо‑ориентированная: гибкость для хранения профилей клиентов, истории взаимодействий, заметок менеджеров.

Реляционная: если важна строгая структура сделок, воронка продаж, финансовые показатели.

1.3. База знаний (нормы, правила, обучение)
Рекомендуемая СУБД: документо‑ориентированная (MongoDB, Elasticsearch) или графовая (Neo4j).

Обоснование:

Иерархическая структура. Данные по компании (отделы, должности, процессы) естественно представляются в виде дерева или графа.

Полнотекстовый поиск. Важен для быстрого поиска по обучающим материалам и нормам.

Гибкость. Возможность хранить документы разных типов (PDF, видео, статьи) с метаданными.

1.4. Логистика и маршрутизация
Рекомендуемая СУБД: графовая (Neo4j, Amazon Neptune).

Обоснование:

Оптимизация маршрутов. Графы идеально моделируют связи между объектами: склады → маршруты → курьеры → адреса доставки.

Быстрый поиск путей. Алгоритмы обхода графа (Dijkstra, A*) находят оптимальные маршруты за доли секунды.

Динамическое обновление. Легко добавлять/удалять узлы (новые объекты, курьеры) и рёбра (дороги, ограничения).

Анализ связей. Можно выявлять узкие места в логистике (перегруженные маршруты, неэффективные склады).

Ответ на вопрос 1.5*
Теоретически можно использовать одну СУБД, но практически это приведёт к компромиссам.

Вариант 1: MongoDB (NoSQL)

Плюсы: гибкость, масштабируемость, поддержка разных типов данных (документы, геоданные для логистики).

Минусы:

сложности с транзакциями между бюджетами и закупками;

неоптимальная производительность для сложных финансовых отчётов;

графовые запросы (логистика) будут медленнее, чем в Neo4j.

Вариант 2: PostgreSQL (реляционная + расширения)

Плюсы:

поддержка JSON (для CRM и базы знаний);

PostGIS (геоданные для логистики);

транзакционность (бюджетирование);

полнотекстовый поиск (база знаний).

Задание 2. Транзакции
2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Приведите ответ в свободной форме.

2.1. Пополнение баланса счёта телефона (ручное пополнение)
Пошаговый процесс успешной транзакции (6 действий):

Шаг 1. Инициирование платежа

Пользователь выбирает способ пополнения (мобильное приложение, сайт, терминал и т. д.).

Вводит номер телефона и сумму пополнения.

Выбирает способ оплаты (банковская карта, электронный кошелёк, наличные в терминале).

Шаг 2. Проверка данных и резервирование средств

Система проверяет корректность введённых данных (номер телефона, сумма).

Если оплата картой — отправляется запрос в банк‑эмитент на авторизацию платежа.

При успешной авторизации средства резервируются на карте пользователя (но не списываются), создаётся временный холд.

Шаг 3. Создание записи о транзакции

В базе данных создаётся запись о транзакции со статусом «В обработке».

Фиксируются: номер телефона, сумма, способ оплаты, время, идентификатор платежа.

Шаг 4. Взаимодействие с оператором связи

Платёжная система отправляет запрос оператору связи на пополнение баланса указанного номера.

Оператор проверяет номер на валидность и доступность для пополнения.

При подтверждении оператор резервирует возможность зачисления суммы.

Шаг 5. Фиксация операции и списание средств

Если оператор подтвердил пополнение, платёжная система списывает зарезервированные средства со счёта пользователя.

Статус транзакции в БД меняется на «Успешно».

В системе оператора связи фиксируется зачисление суммы на баланс номера.

Шаг 6. Уведомление и завершение

Пользователю направляется уведомление об успешном пополнении (SMS, push, email).

В интерфейсе отображается обновлённый баланс номера телефона.

Запись о транзакции сохраняется в истории операций пользователя.


Задание 3. SQL vs NoSQL
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Приведите ответ в свободной форме.

3.1. Пять преимуществ SQL‑систем перед NoSQL
Гарантированная целостность данных (ACID)
SQL‑системы строго соблюдают принципы ACID (атомарность, согласованность, изоляция, долговечность):

каждая транзакция выполняется целиком или не выполняется вовсе (атомарность);

данные всегда остаются в согласованном состоянии (согласованность);

параллельные транзакции не мешают друг другу (изоляция);

после подтверждения транзакции изменения сохраняются навсегда (долговечность).
Это критично для банковских систем, учёта запасов, бронирования и других сценариев, где ошибки недопустимы.

Жёсткая схема данных (схема на записи, schema‑on‑write)
Перед использованием базы данных нужно явно определить структуру таблиц, типы полей и связи между ними. Это даёт:

защиту от случайных ошибок ввода;

чёткое понимание структуры данных для разработчиков;

возможность на уровне БД контролировать типы данных и ограничения (NOT NULL, UNIQUE и т. д.).

Мощный язык запросов (SQL)
SQL — стандартизированный, выразительный язык для работы с данными:

позволяет выполнять сложные выборки с объединениями (JOIN), группировками, подзапросами;

поддерживает агрегатные функции (SUM, AVG, COUNT и т. д.);

подходит для построения аналитических отчётов и сложных бизнес‑правил.

Развитые инструменты для аналитики и отчётности
SQL‑базы данных легко интегрируются с BI‑системами (Power BI, Tableau, Looker) и инструментами ETL. Благодаря строгой схеме и мощному SQL они оптимальны для:

построения дашбордов;

генерации финансовых отчётов;

выполнения сложных аналитических запросов.

Зрелость экосистемы и поддержка
Реляционные СУБД существуют десятилетиями, что обеспечило:

большое количество квалифицированных специалистов;

обширную документацию и сообщества;

богатый выбор инструментов администрирования, мониторинга, резервного копирования;

поддержку со стороны крупных вендоров (Oracle, Microsoft, IBM).


Задание 4. Кластеры
Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу выделено 1000 машин.

На основе какого критерия будете выбирать тип СУБД и какая модель распределённых вычислений здесь справится лучше всего и почему?

Приведите ответ в свободной форме.

Критерий выбора СУБД: тип нагрузки и характер данных. Конкретно нужно учитывать:

тип обработки (OLTP или OLAP);

структуру данных (структурированные, полуструктурированные, графовые);

требования к масштабируемости (горизонтальное/вертикальное);

уровень согласованности (ACID или eventual consistency);

скорость обработки (задержка и пропускная способность).

Оптимальная СУБД в зависимости от задачи:

OLAP‑нагрузки (аналитика, Big Data) → колоночные СУБД: ClickHouse, Apache Druid.

Гибридные нагрузки (OLTP + OLAP) → NewSQL‑системы: CockroachDB, YugabyteDB.

Высокая нагрузка на запись, гибкая схема → NoSQL: Cassandra, ScyllaDB.

Лучшая модель распределённых вычислений: Apache Spark (поверх HDFS или облачного хранилища).

Почему Spark:

масштабируется на 1000+ узлов;

in‑memory обработка — в разы быстрее MapReduce;

единая платформа для:

SQL‑запросов (Spark SQL);

машинного обучения (MLlib);

графовых вычислений (GraphX);

потоковой обработки (Spark Streaming);

интеграция с Kafka, HDFS, S3, Cassandra и др.;

отказоустойчивость: автоматическое восстановление задач при сбоях узлов.
