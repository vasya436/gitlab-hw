### Задание 1. Резервное копирование
Кейс

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

Приведите ответ в свободной форме.

1.1. Восстановление данных в полном объёме за предыдущий день

Для этого случая подходит полное резервное копирование (Full Backup). При полном бэкапе создаётся точная копия всей базы данных, включая все таблицы, индексы, настройки и другие элементы. Это позволяет восстановить базу до состояния на момент создания резервной копии за один шаг, без необходимости дополнительных данных. 

1.2. Восстановление данных за час до предполагаемой поломки

Здесь требуется более частое резервное копирование с минимальным интервалом. Подходят два варианта:

    Инкрементное резервное копирование. Сохраняются только данные, которые были обновлены или добавлены после последнего бэкапа (независимо от его типа). Для восстановления понадобятся полная резервная копия и все последующие инкрементные копии. Этот метод снижает нагрузку на хранилище и ускоряет процесс копирования, но восстановление может занять больше времени. 

    Непрерывное резервное копирование (Continuous Data Protection). Каждая транзакция или изменение в базе данных немедленно реплицируется на резервный сервер. Это позволяет восстановить данные практически в любой момент времени, включая час до сбоя. 

1.3. Моментальное переключение на работающую или починенную базу данных

Да, такой кейс возможен благодаря технологиям высокой доступности и отказоустойчивости. Для этого используются:

    Репликация в реальном времени (синхронная или асинхронная). Данные копируются на вторичный сервер, и при сбое основного сервера происходит автоматическое переключение на резерв. Синхронная репликация гарантирует идентичность данных на резервном сервере, но может влиять на производительность основной системы. Асинхронная позволяет основной базе продолжать работу без ожидания подтверждения от резерва, но небольшая потеря данных при отказе возможна. 

    Кластеризация баз данных. Группа серверов работает так, что при выходе из строя одного из них другой автоматически берёт на себя его функции. Например, технологии Microsoft SQL Server Always On Availability Groups, Oracle Real Application Clusters (RAC) и другие. 

    Отказоустойчивые системы (fault-tolerant systems). Системы, спроектированные так, чтобы при сбое одного компонента другой мгновенно заменял его без потери работы. 

    Системы распределённых баз данных. Данные распределяются между несколькими узлами в разных географических локациях, что позволяет переключаться на работающий узел при сбое другого.

Задание 2. PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

Приведите ответ в свободной форме.

Для резервного копирования и восстановления баз данных PostgreSQL используются утилиты pg_dump (для создания дампа) и pg_restore (для восстановления из дампа).

Пример команды для резервного копирования с помощью pg_dump

Базовая команда для создания текстового дампа базы данных:

pg_dump -U имя_пользователя имя_базы > backup.sql

Пример команды для восстановления с помощью pg_restore

Утилита pg_restore используется для восстановления базы данных из архива, созданного pg_dump. 

Пример восстановления всей базы данных:

pg_restore -U postgres -d dvdrental D:/backup/dvdrental.tar

Автоматизация процесса

Да, процесс резервного копирования и восстановления можно автоматизировать. Для этого обычно используют скрипты и планировщики задач (например, cron в Unix-подобных системах или планировщик заданий в Windows). 

При автоматизации важно учитывать такие аспекты, как безопасность (хранение паролей, ограничение доступа), проверка целостности резервных копий и настройка ротации (удаление устаревших копий). 

Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL.

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

Приведите ответ в свободной форме.

Инкрементное резервное копирование в MySQL обычно выполняется с использованием бинарных логов (binary logs) или утилиты mysqlbackup из MySQL Enterprise Backup (платный продукт).


Инкрементное копирование с помощью бинарных логов

Для работы с бинарными логами необходимо предварительно включить их в конфигурации MySQL, установив параметр log_bin. Например, в файле конфигурации (my.cnf или mysqld.cnf) можно добавить или изменить строки: 

log_bin = /var/log/mysql/mysql-bin.log
expire_logs_days = 10

После сохранения изменений нужно перезапустить службу MySQL. 

Процесс инкрементного бэкапа:

    Сначала необходимо выполнить полное резервное копирование с помощью mysqldump и инициализировать бинарные логи:

    mysqldump -u root -p --all-databases --single-transaction --flush-logs --master-data=2 > full_backup.sql

После этого все изменения в базе данных будут записываться в бинарные логи. Для создания инкрементного бэкапа нужно скопировать новые бинарные логи, которые появились после полного бэкапа. 

Для восстановления из инкрементного бэкапа сначала восстанавливают полную копию, а затем применяют изменения из бинарных логов с помощью утилиты mysqlbinlog

Инкрементное копирование с помощью mysqlbackup

Утилита mysqlbackup (MySQL Enterprise Backup) позволяет создавать инкрементные бэкапы с использованием отслеживания изменённых страниц (page tracking) или указания базового бэкапа. 

Пример команды с использованием параметра --incremental-base:

mysqlbackup --defaults-file=/home/dbadmin/my.cnf \
--incremental=optimistic \
--incremental-base=history:last_backup \
--backup-dir=/home/dbadmin/temp_dir \
--backup-image=incremental_image1.bi \
backup-to-image

Когда репликация предпочтительнее резервного копирования

Репликация в MySQL -это процесс копирования данных с одного сервера (мастера) на один или несколько других серверов (слейвов). Она имеет ряд преимуществ перед традиционным резервным копированием:

    Высокая доступность. При сбое основного сервера можно быстро переключиться на реплику, минимизируя время простоя. В то время как восстановление из резервной копии может занять значительное время, особенно для больших баз данных.
    Распределение нагрузки. Реплики можно использовать для операций чтения (например, отчётов, аналитических запросов), что снижает нагрузку на основной сервер. Это особенно полезно в высоконагруженных системах. 
    Географическое распределение. Реплики можно размещать в разных географических регионах, улучшая время отклика для пользователей в разных частях мира. 

Тестирование. Реплики позволяют создавать тестовые среды, которые точно повторяют производственную среду, без риска для рабочих данных. 

Непрерывное обновление данных. Репликация поддерживает копии данных постоянно актуальными, в отличие от резервного копирования, которое выполняется периодически. 
Однако у репликации есть и недостатки:

    требует более сложной настройки и мониторинга;
    может увеличивать нагрузку на сеть и требовать стабильного соединения;
    в случае асинхронной репликации возможны временные расхождения данных между мастером и слейвами. 

Оптимальный подход - использовать репликацию для обеспечения высокой доступности и распределения нагрузки, а резервное копирование — для защиты от потери данных из-за аппаратных сбоев, атак или ошибок администрирования. Эти технологии дополняют друг друга.
